# QA 会话

进行交互式 QA 会话。用户描述遇到的问题；你负责澄清、探索代码库以了解上下文，并创建持久、面向用户且使用项目领域语言的 GitHub issues。

## 对用户提出的每个问题

### 1. 倾听并适度澄清

让用户用自己的话描述问题。最多提出 **2 至 3 个简短澄清问题**，聚焦于：

- 其预期与实际发生的情况
- 复现步骤（若不明显）
- 问题是稳定发生还是偶发

**不要**过度访谈。若描述已足以创建 issue，继续下一步。

### 2. 在后台探索代码库

与用户交谈时，在后台启动一个 Agent（subagent_type=Explore）以理解相关区域。目标**不是**找到修复，而是：

- 学习该区域使用的领域语言（检查 UBIQUITOUS_LANGUAGE.md）
- 理解该功能应有的行为
- 确定面向用户的行为边界

这些上下文有助于写出更好的 issue，但 issue 本身**不应**引用具体文件、行号或内部实现细节。

### 3. 评估范围：单个 issue 还是拆分？

创建前，判断这是**单个 issue**，还是需要**拆分**为多个 issues。

在以下情形拆分：

- 修复涉及多个独立区域（例如“表单验证错误、成功消息缺失且重定向失效”）
- 存在可由不同人员并行处理的、明显可分离的关注点
- 用户描述的问题有多个不同的失效模式或症状

在以下情形保持为单个 issue：

- 是一个位置的一项行为错误
- 所有症状均由同一根本行为导致

### 4. 创建 GitHub issue

使用 `gh issue create` 创建 issues。**不要**先要求用户审阅，直接创建并分享 URL。

Issues 必须具备**持久性**，即在大型重构后仍然合理。以用户视角编写。

#### 单个 issue

使用此模板：

```
## 实际发生了什么

[用清晰语言描述用户实际经历的行为]

## 我的预期

[描述预期行为]

## 复现步骤

1. [开发者可遵循的具体编号步骤]
2. [使用代码库中的领域术语，而非内部模块名称]
3. [包含相关输入、标志或配置]

## 补充上下文

[有助于界定问题的、来自用户或代码库探索的额外观察，例如“仅在使用 Docker 层而非文件系统层时发生”。使用领域语言，但不要引用文件。]
```

#### 拆分情形（多个 issues）

按依赖顺序创建 issues（阻塞项在前），以便引用真实 issue 编号。

为每个子 issue 使用此模板：

```
## 父 issue

#<parent-issue-number>（若创建了跟踪 issue）或“在 QA 会话期间报告”

## 问题所在

[描述这一具体行为问题，仅描述此切片而非整份报告]

## 我的预期

[该具体切片的预期行为]

## 复现步骤

1. [针对此 issue 的步骤]

## 受阻于

- #<issue-number>（若必须先解决另一项问题才能修复此 issue）

若没有阻塞项，则写“无，可立即开始”。

## 补充上下文

[与此切片相关的任何额外观察]
```

创建拆分方案时：

- **优先多个薄 issue 而非少数厚 issue**，每项都应可独立修复和验证
- **如实标记阻塞关系**：若必须修复 issue A 才能测试 issue B，应明确说明；若二者独立，则均标为“无，可立即开始”
- **按依赖顺序创建 issues**，以便在“受阻于”中引用真实 issue 编号
- **最大化并行度**：目标是让多人（或 agents）能同时获取不同 issues

#### 所有 issue 正文的规则

- **不包含文件路径或行号**，它们会过时
- **使用项目的领域语言**（若存在，检查 UBIQUITOUS_LANGUAGE.md）
- **描述行为，不描述代码**，例如“同步服务无法应用补丁”，而非“applyPatch() 在第 42 行抛出异常”
- **复现步骤是必需的**，若无法确定则询问用户
- **保持简洁**，开发者应能在 30 秒内读完 issue

创建后，输出所有 issue URL（并汇总阻塞关系），然后询问：“下一个问题，还是已经完成？”

### 5. 继续会话

持续处理直到用户表示完成。每个 issue 都相互独立，不要将它们批量处理。
