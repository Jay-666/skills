# 通用语言

从当前对话中提取领域术语并将其形式化为一致的术语表，保存到本地文件。

## 流程

1. **扫描对话**，找出与领域相关的名词、动词和概念
2. **识别问题**：
   - 同一个词用于不同概念（歧义）
   - 不同词用于同一个概念（同义词）
   - 含糊或多义术语
3. 提出带有明确取舍的**规范术语表**
4. 按以下格式将其写入工作目录中的 `UBIQUITOUS_LANGUAGE.md`
5. 在对话中内联输出**摘要**

## 输出格式

按以下结构编写 `UBIQUITOUS_LANGUAGE.md` 文件：

```md
# 通用语言

## 订单生命周期

| 术语        | 定义                                              | 应避免的别名      |
| ----------- | ------------------------------------------------------- | --------------------- |
| **订单**   | 客户购买一个或多个商品的请求      | 采购、交易 |
| **发票** | 交付后向客户发送的付款请求 | 账单、付款请求 |

## 人员

| 术语         | 定义                                  | 应避免的别名       |
| ------------ | ------------------------------------------- | ---------------------- |
| **客户** | 下订单的个人或组织 | 客户端、买方、账户 |
| **用户**     | 系统中的认证身份    | 登录、账户         |

## 关系

- 一张**发票**恰好属于一位**客户**
- 一张**订单**会产生一张或多张**发票**

## 示例对话

> **开发者：** “当一位**客户**下了一张**订单**时，会立即创建**发票**吗？”
> **领域专家：** “不会。只有确认了**履约**才会生成**发票**。如果商品分为多次**发货**，一张**订单**可以产生多张**发票**。”
> **开发者：** “那么如果某次**发货**在派送前取消，就不会有对应的**发票**吗？”
> **领域专家：** “完全正确。**发票**的生命周期与**履约**相关，而非与**订单**相关。”

## 已标记的歧义

- “账户”曾同时指代**客户**和**用户**。两者是不同概念：**客户**下订单，**用户**则是可能代表也可能不代表**客户**的认证身份。
```

## 规则

- **作出明确取舍。** 当同一概念存在多个词时，选择最佳术语，并将其他词列为应避免的别名。
- **明确标记冲突。** 若术语在对话中使用有歧义，在“已标记的歧义”章节指出，并给出明确建议。
- **仅纳入与领域专家相关的术语。** 除非模块或类名在领域语言中有含义，否则跳过。
- **保持定义简洁。** 最多一句。定义它“是什么”，而非“做什么”。
- **展示关系。** 使用加粗的术语名称，并在明显时表达基数。
- **仅纳入领域术语。** 跳过通用编程概念（数组、函数、端点），除非它们具有领域专属含义。
- 当自然出现聚类时，**将术语分组为多个表格**（例如按子域、生命周期或参与者）。每组使用独立标题和表格。若所有术语属于一个内聚领域，一个表格即可，不要强行分组。
- **编写示例对话。** 用开发者与领域专家间的简短对话（3 至 5 轮）展示术语如何自然交互。对话应澄清相关概念的边界并精确使用术语。

## 示例对话

> **开发者：** “如何在不使用 Docker 的情况下测试**同步服务**？”

> **领域专家：** “提供**文件系统层**而不是 **Docker 层**。它实现相同的**沙箱服务**接口，但使用本地目录作为**沙箱**。”

> **开发者：** “所以 **sync-in** 仍会创建并解包一个**包**吗？”

> **领域专家：** “没错。**同步服务**并不知道它正在与哪一层交互。它调用 `exec` 和 `copyIn`，而**文件系统层**只是将它们作为本地 shell 命令运行。”

</example>

## 再次运行

在同一对话中再次调用时：

1. 阅读现有的 `UBIQUITOUS_LANGUAGE.md`
2. 纳入后续讨论中的所有新术语
3. 若理解发生演变，则更新定义
4. 重新标记所有新歧义
5. 重写示例对话以纳入新术语
