description: 通过访谈用户创建包含微小提交的详细重构计划，然后将其创建为 GitHub issue。适用于用户希望规划重构、创建重构 RFC，或将重构拆分为安全的渐进步骤时。
---

当用户希望创建重构请求时调用此 skill。应执行以下步骤；若认为没有必要，可以跳过其中的步骤。

1. 请用户详细描述其希望解决的问题，以及所有可能的解决思路。

2. 探索仓库以验证其说法，并了解代码库当前状态。

3. 询问其是否考虑过其他选项，并向其提出替代选项。

4. 就实现方案访谈用户。应极其详细和全面。

5. 明确实现的精确范围。确定计划改变的内容和计划不改变的内容。

6. 检查代码库中该区域的测试覆盖率。若测试覆盖不足，询问用户的测试计划。

7. 将实现拆分为微小提交的计划。牢记 Martin Fowler 的建议：“让每个重构步骤尽可能小，这样你始终能看到程序仍在工作。”

8. 使用重构计划创建 GitHub issue。issue 描述使用以下模板：

<refactor-plan-template>

## 问题陈述

从开发者视角描述其面临的问题。

## 解决方案

从开发者视角描述问题的解决方案。

## 提交

一份很长且详细的实现计划。以清晰的自然语言编写，将实现拆分为尽可能微小的提交。每个提交都应使代码库保持可用状态。

## 决策文档

列出已作出的实现决策，可包括：

- 将要构建或修改的模块
- 将要修改的模块接口
- 开发者给出的技术澄清
- 架构决策
- Schema 变更
- API 契约
- 具体交互

**不要**包含具体文件路径或代码片段，它们可能很快过时。

## 测试决策

列出已作出的测试决策，包括：

- 好测试的定义（仅测试外部行为，而非实现细节）
- 将要测试的模块
- 测试的既有范例（即代码库中的相似测试）

## 范围外内容

说明不属于本次重构范围的内容。

## 补充说明（可选）

有关该重构的其他说明。

</refactor-plan-template>
