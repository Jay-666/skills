# 设计接口

基于《软件设计哲学》中的“设计两次”：第一个想法不太可能是最佳方案。生成多个截然不同的设计，然后进行比较。

## 工作流

### 1. 收集需求

在设计前，先了解：

- [ ] 此模块要解决什么问题？
- [ ] 调用方是谁？（其他模块、外部用户、测试）
- [ ] 关键操作有哪些？
- [ ] 有哪些约束？（性能、兼容性、既有模式）
- [ ] 哪些内容应隐藏在内部，哪些应暴露？

询问：“此模块需要做什么？谁会使用它？”

### 2. 生成设计（并行子 agent）

使用 Task 工具同时启动 3 个以上子 agent。每个都必须产出**截然不同**的方法。

```
每个子 agent 的提示词模板：

为以下对象设计接口：[模块描述]

需求：[已收集的需求]

此设计的约束：[为每个 agent 分配不同约束]
- Agent 1：“最小化方法数量，最多以 1 至 3 个方法为目标”
- Agent 2：“最大化灵活性，支持多种使用场景”
- Agent 3：“针对最常见场景优化”
- Agent 4：“从[特定范式或库]中获得灵感”

输出格式：
1. 接口签名（类型或方法）
2. 使用示例（调用方如何使用）
3. 此设计在内部隐藏了什么
4. 该方法的权衡
```

### 3. 呈现设计

展示每个设计的：

1. **接口签名**：类型、方法、参数
2. **使用示例**：调用方在实践中实际如何使用
3. **隐藏内容**：保留在内部的复杂性

按顺序呈现设计，让用户在比较前充分理解每种方法。

### 4. 比较设计

展示所有设计后，从以下方面比较：

- **接口简单性**：更少的方法、更简单的参数
- **通用性与专用性**：灵活性与聚焦度
- **实现效率**：接口形态是否支持高效的内部实现？
- **深度**：小接口隐藏显著复杂性（好），与大接口配合单薄实现（差）
- **正确使用的容易度**与**误用的容易度**

用文字而非表格讨论取舍。突出各设计分歧最大的地方。

### 5. 综合

最佳设计通常结合多个选项的洞见。询问：

- “哪个设计最适合你的主要使用场景？”
- “其他设计中是否有值得纳入的元素？”

## 评估标准

出自《软件设计哲学》：

**接口简单性**：方法越少、参数越简单，越容易学习和正确使用。

**通用性**：无需改动即可应对未来使用场景。但要警惕过度泛化。

**实现效率**：接口形态是否允许高效实现？还是迫使内部结构别扭？

**深度**：小接口隐藏显著复杂性就是深模块（好）。大接口配合单薄实现就是浅模块（应避免）。

## 反模式

- 不要让子 agent 产出相似设计，要强制保证明显差异
- 不要跳过比较，价值在于对比
- 不要实现，此处只讨论接口形态
- 不要依据实现工作量进行评估
