# 向导

**向导**是一个 bash 脚本，逐步引导人完成手动流程。这类流程手动执行很繁琐，每次向 AI 重新解释也很繁琐。它会打开每个 URL，准确说明点击和复制什么，捕获值，将值写入其应在的位置（`.env`、GitHub secrets），在每个阶段确认，并展示剩余工作量。它可配置第三方服务、运行一次性迁移，或将项目从一个状态转移到另一个状态。

良好的 UX 已由 [template.sh](template.sh) 解决：包含带剩余时间的进度、确认关卡、跨平台 URL 打开（包括 WSL）、隐藏式秘密输入、幂等 `.env` upsert、`gh secret` 或 `gh variable` 写入，以及结尾摘要。**你的工作仅是确定流程范围并编写阶段。** 每个向导中 `STAGES` 标记上方的库都相同，统一性正是其目的，绝不可手动编辑它。

向导默认是短暂的：为一次运行构建，保存到临时目录或 `scripts/` 路径，任务完成后删除。仅当用户希望在仓库中保留可重复的配置路径时才提交它。

## 流程

### 1. 确定流程范围

梳理人必须执行的每个手动步骤和过程中捕获的每个值。先读取仓库，不要直接询问：

- 配置场景：`.env`、`.env.example`、`.env.*`、`README`、`docker-compose*`、框架配置和 `.github/workflows/*`（每个 `secrets.*` 或 `vars.*` 引用都是向导必须产出的值）。
- 迁移或转换场景：当前状态、目标状态及二者之间不可逆的动作。

随后向用户展示有序的阶段列表及每阶段产出的值，并确认。用户可添加、删除或重新排序。

**完成标准：** 每个阶段均按顺序命名；对每个捕获值，都知道 (a) 人从何处获取，(b) 写入哪里（`.env`、GitHub secret、两者或都不写，某些阶段纯粹执行动作），以及 (c) 它是秘密值（隐藏输入）还是公开值。

### 2. 映射每个阶段的路径

为每个阶段写明人要遵循的精确路径：打开哪个 URL、在那里做什么、值显示在哪里、填写哪个变量，例如“控制台 → 开发者 → API 密钥 → 显示测试密钥 → 复制”。若并不真正了解当前 UI 或准确命令，应明确说明并询问用户或检查文档，绝不编造可能不存在的步骤。

**完成标准：** 每个阶段都可追溯至陌生人也能遵循的具体说明。

### 3. 编写向导

将 `template.sh` 复制到目标路径。将示例阶段替换为每步骤一个 `stage`，按依赖顺序排列。使用库辅助函数 `stage`、`say` 或 `step`、`open_url`、`ask` 或 `ask_secret`、`write_env`、`set_secret` 或 `set_var`、`pause` 或 `confirm`，并将 `TOTAL_STAGES` 和 `TOTAL_MINUTES` 设为诚实的估计值（它们驱动剩余时间显示）。

保持模板设定的标准：在询问 URL 上的值前先打开 URL；任何秘密值使用 `ask_secret`；每个持久化值使用 `write_env`；仅为 CI 实际需要的值使用 `set_secret`；任何不可逆动作前使用 `confirm`。每个 `stage` 都会清屏，因此只显示当前步骤，应使一个阶段只聚焦一项任务，避免人需要的内容滚出屏幕。不要触碰标记上方的库。

### 4. 验证并交接

- 执行 `bash -n <script>`；若可用则运行 `shellcheck`。
- 执行 `chmod +x <script>`。
- 不要亲自端到端运行它，它会打开浏览器并阻塞在人工输入处。改为静态追踪：第 1 步中的每个值都已捕获并落在第 1 步所说的位置，且每个 `set_secret` 名称都准确匹配 CI 中的 `secrets.*` 引用。
- 告知用户如何运行它。若它是可重复的配置路径，则提交它并从 README 链接，让下一位使用者运行脚本而非询问 AI。
