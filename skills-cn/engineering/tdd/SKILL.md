# 测试驱动开发

TDD 是红 → 绿循环。此 skill 提供让该循环产出值得保留的测试所需的参考：好测试的定义、测试位置、反模式和循环规则。每个章节都适用于每次循环，应在循环前和循环中查阅，而非结束后。

探索代码库时，阅读 `CONTEXT.md`（若存在），使测试名称和接口词汇符合项目领域语言，并遵守涉及区域的 ADR。

## 什么是好测试

测试应通过公共接口验证行为，而非实现细节。代码可以完全变化，测试不应随之变化。好测试读起来像规格说明，例如“用户可以使用有效购物车结账”准确表达了已具备的能力；它不关心内部结构，因此能够经受重构。

示例见 [tests.md](tests.md)，mock 指南见 [mocking.md](mocking.md)。

## 边界：测试的位置

**边界**是进行测试的公共边界：你无需深入内部即可观察行为的接口。测试应位于边界处，绝不直接针对内部实现。

**仅在预先约定的边界处测试。** 编写任何测试前，写下将要测试的边界并与用户确认。不得在未确认的边界处编写测试。你无法测试所有内容，预先约定边界才能让测试工作量落在关键路径和复杂逻辑上，而非每个边缘情况。

询问：“公共接口是什么，应测试哪些边界？”

## 反模式

- **与实现耦合**：mock 内部协作者、测试私有方法，或通过旁路验证（例如查询数据库而不使用接口）。判断信号是：重构后行为未变，测试却失败。
- **同义反复**：断言以与代码相同的方式重新计算期望值（`expect(add(a, b)).toBe(a + b)`、以相同方式手工生成的快照、断言常量等于自身），因此它会按构造通过，永远无法与代码产生分歧。期望值必须来自独立的事实来源，例如已知正确的字面量、完整示例或规格说明。
- **水平切片**：先写完全部测试，再写完全部实现。批量测试验证的是*想象中的*行为：测试的是事物的*形状*而非面向用户的行为，测试会对真实变更失去敏感性，并且会在理解实现前就锁定测试结构。应改用**垂直切片**：一个测试 → 一个实现 → 重复；每个测试都是对上次循环经验作出响应的**垂直切片**。

## 循环规则

- **先红后绿。** 先编写失败测试，再仅编写使其通过所需的代码。不要预先推测未来测试，也不要添加投机性功能。
- **一次一个切片。** 每次循环一个边界、一个测试、一个最小实现。
- **重构不是循环的一部分。** 它属于审查阶段（参见 `code-review` skill），而非红 → 绿实现循环。
