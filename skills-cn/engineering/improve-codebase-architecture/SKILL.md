# 改进代码库架构

发现架构摩擦，并提出**加深机会**，即将浅模块转化为深模块的重构。目标是可测试性和 AI 可导航性。

此命令受项目领域模型*启发*，并建立在共享设计词汇之上：

- 运行 `/codebase-design` skill 获取架构词汇（**模块**、**接口**、**深度**、**边界点**、**适配器**、**杠杆**、**局部性**）和原则（删除测试、“接口是测试表面”“一个适配器 = 假想边界点，两个 = 真实边界点”）。在每项建议中准确使用这些术语，不要偏移为“组件”“服务”“API”或“boundary”。
- `CONTEXT.md` 中的领域语言为好的边界点命名；`docs/adr/` 中的 ADR 记录了本命令不应重新争论的决策。

## 流程

### 1. 探索

**扫描前先确定范围，遵循 YAGNI。** 加深模块通过让未来对它的变更更容易而产生收益，因此应额外关注代码库中最近变化的部分。在查看前决定*查看哪里*：

- 若用户指定方向，例如模块、子系统或痛点，采用它并跳过下方推断。
- 否则，回溯一段提交历史（`git log --oneline`）以找出代码库热点，即反复出现的文件和区域，让这些路径优先吸引注意力。若变更分散且没有明显热点，则扩大范围。

先阅读项目领域术语表（`CONTEXT.md`）和涉及区域的所有 ADR。

然后使用带 `subagent_type=Explore` 的 Agent 工具遍历代码库。不要遵循僵化的启发式规则，应自然探索并记录感受到摩擦的位置：

- 在哪里理解一个概念需要在许多小模块之间跳转？
- 在哪里模块是**浅**的，即接口几乎与实现一样复杂？
- 在哪里仅为了可测试性提取了纯函数，但真实缺陷隐藏在调用方式中（没有**局部性**）？
- 在哪里紧密耦合模块跨越其边界点泄漏？
- 代码库哪些部分未测试，或难以通过当前接口测试？

对怀疑浅的任何内容应用**删除测试**：删除它会集中复杂性，还是仅移动复杂性？“是，会集中”是你想要的信号。

### 2. 将候选项呈现为 HTML 报告

向操作系统临时目录写入一个自包含 HTML 文件，使任何内容都不落入仓库。从 `$TMPDIR` 解析临时目录，回退到 `/tmp`（Windows 上为 `%TEMP%`），写入 `<tmpdir>/architecture-review-<timestamp>.html`，使每次运行都有新文件。为用户打开它，Linux 使用 `xdg-open <path>`、macOS 使用 `open <path>`、Windows 使用 `start <path>`，并告知绝对路径。

报告通过 CDN 使用 **Tailwind** 进行布局和样式，并在图、流程或序列能可靠表达结构时通过 CDN 使用 **Mermaid** 绘图。混合使用 Mermaid 和手工 CSS 或 SVG 视觉效果：当关系是图形形态（调用图、依赖、序列）时使用 Mermaid；当需要更具编辑感的内容（质量图、横截面、折叠动画）时使用手工 div 或 SVG。每个候选项都要有**前后对比可视化**。应以视觉为主。

为每个候选项渲染一张卡片，包含：

- **文件**：涉及哪些文件或模块
- **问题**：当前架构为何造成摩擦
- **方案**：用自然语言描述会发生什么变更
- **收益**：以局部性和杠杆解释，并说明测试将如何改进
- **前后对比图**：并排、定制绘制，展示浅层状态和加深后状态
- **建议强度**：`Strong`、`Worth exploring` 或 `Speculative` 之一，渲染为徽章

在报告结尾添加**首要建议**章节，说明你会优先处理哪个候选项以及原因。

**领域使用 CONTEXT.md 词汇，架构使用 `/codebase-design` 词汇。** 若 `CONTEXT.md` 定义“订单”，应说“订单接收模块”，而非“FooBarHandler”或“订单服务”。

**ADR 冲突**：若候选项与现有 ADR 矛盾，仅当摩擦真实到足以重新审视 ADR 时才提出。应在卡片中明确标记，例如警告提示：_“与 ADR-0007 矛盾，但值得重新开启，因为……”_。不要列出 ADR 禁止的每一项理论重构。

完整 HTML 脚手架、图形模式和样式指南见 [HTML-REPORT.md](HTML-REPORT.md)。

暂时**不要**提出接口。文件写入后，询问用户：“你想探索其中哪个？”

### 3. 深入追问循环

用户选择候选项后，运行 `/grilling` skill 与其一起走过决策树，包括约束、依赖、加深模块的形态、边界点后隐藏的内容和可保留的测试。

随着决策明确，副作用会内联发生，运行 `/domain-modeling` skill 以保持领域模型最新：

- **以 `CONTEXT.md` 中没有的概念命名加深模块？** 将术语加入 `CONTEXT.md`。若文件不存在则按需创建。
- **在对话中明确模糊术语？** 立即更新 `CONTEXT.md`。
- **用户以关键理由拒绝候选项？** 提出 ADR，表述为：“要我将此记录为 ADR，以便未来架构审查不再重新建议它吗？”仅当未来探索者确实需要此理由来避免再次提出相同建议时才提出，跳过短暂理由（“现在不值得”）和不言自明的理由。
- **想探索加深模块的替代接口？** 运行 `/codebase-design` skill，并使用其“设计两次”并行子 agent 模式。
