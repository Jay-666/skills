# 原型

原型是**回答一个问题的一次性代码**。问题决定其形态。

## 选择分支

确定要回答哪个问题，可以从用户提示词、周围代码判断，或在用户可联系时询问：

- **“此逻辑或状态模型合理吗？”** → [LOGIC.md](LOGIC.md)。构建一个微型交互式终端应用，使状态机经历纸面上难以推理的场景。
- **“它应该是什么样子？”** → [UI.md](UI.md)。在单一路由上生成多个截然不同的 UI 变体，通过 URL 搜索参数和浮动底部栏切换。

两个分支产出的工件截然不同，选错会浪费整个原型。若问题确有歧义且无法联系用户，默认选择更符合周围代码的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部说明该假设。

## 同时适用于两者的规则

1. **从第一天起就是一次性内容，并明确标记。** 将原型代码放在实际使用位置附近（所原型化模块或页面旁），使上下文显而易见；但命名应让随意阅读者知道它是原型而非生产代码。对于一次性 UI 路由，遵守项目已有的路由约定，不要发明新的顶层结构。
2. **一个启动命令。** 使用项目现有任务运行器支持的任意方式，如 `pnpm <name>`、`python <path>`、`bun <path>`。用户必须无需思考即可启动它。
3. **默认不持久化。** 状态位于内存中。持久化是原型*正在验证*的内容，而非它应依赖的内容。若问题明确涉及数据库，使用带有清晰“PROTOTYPE — wipe me”名称的临时数据库或本地文件。
4. **跳过打磨。** 不写测试、不做超出原型*可运行*所需的错误处理、不加抽象。目标是快速学习。
5. **展示状态。** 每次动作后（逻辑）或每次变体切换时（UI），打印或渲染完整相关状态，让用户看清变化。
6. **完成后捕获。** 将任何已验证的决策纳入真实代码，然后把原型本身作为**一手来源**捕获：提交到独立于 main 的一次性分支，并在实现 issue 中留下指向该分支的上下文指针。还要在 issue 或提交中捕获答案，即结论和它解决的问题。main 分支仅保留已验证的决策。
