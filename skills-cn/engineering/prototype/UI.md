# UI 原型

在单一路由上生成**多个截然不同的 UI 变体**，可从浮动底部栏切换。用户在浏览器中切换变体，选择一个（或从每个中借用部分），然后丢弃其余内容。

若问题涉及逻辑或状态而非外观，则分支错误。使用 [LOGIC.md](LOGIC.md)。

## 何时使用这种形态

- “此页面应该是什么样子？”
- “我想在确定前看到这个仪表盘的几个选项。”
- “为设置屏幕尝试不同布局。”
- 用户原本会花一天在脑中从三个模糊 mockup 中选择的任何时候。

## 两种子形态：强烈优先子形态 A

当 UI 原型**紧贴应用其余部分**时更容易判断，包括真实页头、真实侧栏、真实数据和真实密度。独立的一次性路由是一个真空，所有变体在隔离中看起来都不错。只要有合理的现有页面承载变体，默认使用子形态 A。仅当原型确实没有附近归宿时才选择子形态 B。

### 子形态 A：调整现有页面（首选）

路由已存在。变体渲染在**同一路由**上，通过 `?variant=` URL 搜索参数控制。现有数据获取、参数和认证全部保留，只交换渲染内容。这是默认选择，除非有具体理由不这样做。

若原型对应的内容尚没有页面，但*自然应位于某个页面内*（仪表盘的新章节、设置屏幕上的新卡片、现有流程中的新步骤），它仍属于子形态 A。将变体挂载在宿主页面内。

### 子形态 B：新页面（最后手段）

仅当原型化内容确实没有可嵌入的现有页面时使用，例如完全新的顶层表面，或无法合理嵌入任何位置的流程。

遵循项目已有路由约定创建**一次性路由**，不要发明新的顶层结构。命名应使其明显是原型，例如在路径或文件名中包含 `prototype`。使用相同的 `?variant=` 模式。

确定使用子形态 B 前，进行合理性检查：真的没有现有页面可嵌入吗？空路由会隐藏已填充路由会暴露的设计问题。

两种子形态中的浮动底部栏相同。

## 流程

### 1. 陈述问题并选择 N

默认使用**3 个变体**。超过 5 个不再是截然不同，而开始成为噪声，应以此为上限。

在原型位置或文件顶部注释中用一行写下计划：

> “设置页面的三个变体，通过 `?variant=` 在现有 `/settings` 路由上切换。”

无论用户是否在场提出异议，这都适用。

### 2. 生成截然不同的变体

起草每个变体，并确保每个都符合：

- 页面的目的和它可访问的数据。
- 项目的组件库或样式系统（TailwindCSS、shadcn、MUI、普通 CSS 等）。
- 清晰导出的组件名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**，包括不同布局、不同信息层级、不同主要可供性，而不只是颜色不同。三个略微调整的卡片网格不是 UI 原型，而是壁纸。若两个草稿过于相似，以明确的“不使用卡片网格”指示重做其中一个。

### 3. 将它们接在一起

在该 route 上创建一个单独的切换组件：

```tsx
// 伪代码：适配项目框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：将所有现有数据获取保留在切换器上方，仅渲染子树因变体而变化。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同切换器。

### 4. 构建浮动切换器

位于屏幕底部中央的小型固定定位栏，包含三部分：

- **左箭头**：循环到上一个变体（环绕）。
- **变体标签**：展示当前变体键，若变体导出名称也展示名称，例如 `B — Sidebar layout`。
- **右箭头**：循环到下一个变体（环绕）。

行为：

- 点击箭头更新 URL 搜索参数（使用框架路由器，例如 Next 的 `router.replace`、React Router 的 `navigate` 等），使变体可分享且重载稳定。
- 键盘：`←` 和 `→` 箭头键也可循环。聚焦 `<input>`、`<textarea>` 或 `[contenteditable]` 时不要拦截箭头键。
- 在视觉上与页面不同，例如高对比度胶囊、细微阴影，使其明显不是正在评估的设计一部分。
- 在生产构建中隐藏，通过 `process.env.NODE_ENV !== 'production'` 或等效检查控制，使意外合并的原型不会将栏交付给用户。

将切换器放在单个共享组件中，使两个子形态均可复用。将其放在项目共享 UI 所在位置。

### 5. 交给用户

展示 URL（以及 `?variant=` 键）。用户有空时会切换浏览。通常有趣的反馈是**“我想要 B 的页头加上 C 的侧栏”**，这才是他们实际想要的设计。

### 6. 捕获答案并清理

某个变体获胜后，捕获答案，即哪个变体以及原因，然后按 [SKILL](SKILL.md) 所述捕获原型。将胜者纳入真实代码，将其余内容移至一次性分支而非 main：

- **子形态 A**：将胜者纳入现有页面，从 main 移除失败变体和切换器。
- **子形态 B**：将获胜变体提升为真实路由，从 main 移除一次性路由和切换器。

完整变体集合是一手来源，因此应落在一次性分支而非垃圾箱，留在 main 分支的变体组件和切换器会快速腐朽并使下一位读者困惑。

## 反模式

- **仅颜色或文案不同的变体。** 这是微调而非原型，真实变体应在结构上存在分歧。
- **在变体之间共享过多代码。** 共享 `<Header>` 没问题，共享 `<Layout>` 会违背目的。每个变体应可自由抛弃布局。
- **将变体接入真实修改操作。** 只读原型没有问题。若变体需要修改，指向 stub，问题是“它应该是什么样子”，而非“后端是否工作”。
- **将原型直接提升到生产。** 变体代码在原型约束下编写（无测试、最小错误处理）。纳入真实代码时应正确重写。
