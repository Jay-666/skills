# 逻辑原型

一个允许用户手动驱动状态模型的微型交互式终端应用。当问题涉及**业务逻辑、状态转换或数据形状**时使用，它们在纸面上看似合理，但只有通过真实场景推进后才会显得错误。

## 何时使用这种形态

- “我不确定此状态机是否处理了先 X 后 Y 的边缘情况。”
- “此数据模型真的能让我表示……的情形吗？”
- “我想在编写 API 前感受一下它应该是什么样子。”
- 用户想要**按按钮并观察状态变化**的任何情形。

若问题是“它应该是什么样子”，则分支错误。使用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

编写代码前，写下要原型化的状态模型和问题。在原型 README 或文件顶部注释中使用一段话。回答错误问题的逻辑原型纯属浪费，应明确问题，以便之后检查，无论用户现在观察还是之后无人值守地返回。

### 2. 选择语言

使用宿主项目使用的语言。若项目没有明显运行时（例如文档仓库），则询问。

遵守项目现有工具约定，不要仅为原型添加新的包管理器或运行时。

### 3. 将逻辑隔离到可移植模块中

将实际逻辑，即回答问题的部分，置于小型纯接口后，使之后可将其提取并放入真实代码库。围绕它的 TUI 是一次性的，逻辑模块不应如此。

正确形态取决于问题：

- **纯 reducer**：`(state, action) => state`。适合动作是离散事件、状态是单个值的情形。
- **状态机**：显式状态和转换。适合“当前哪些动作甚至是合法的”是问题一部分的情形。
- 针对普通数据类型的**一小组纯函数**。适合没有隐式当前状态、只有转换的情形。
- 当逻辑确实拥有持续的内部状态时，使用拥有清晰方法表面的**类或模块**。

选择最符合所问问题的形态，而*不是*最容易接入 TUI 的形态。保持纯净：没有 I/O、没有终端代码、没有用于控制流的 `console.log`。TUI 导入并调用它，任何内容均不反向流动。

这使原型在其寿命之外仍有用：问题回答后，经过验证的 reducer、状态机或函数集可自行提取到真实模块中。

### 4. 构建展示状态的最小 TUI

将它构建为**轻量 TUI**，每个 tick 都清屏（`console.clear()`、`print("\033[2J\033[H")` 或等效方式）并重新渲染整个帧。用户应始终看到一个稳定视图，而非不断增长的滚动历史。

每个帧依次包含两个部分：

1. **当前状态**，以美观且便于比较差异的方式打印（每行一个字段，或格式化 JSON）。字段名称或章节标题使用**粗体**，较不重要的上下文（时间戳、ID、派生值）使用**暗色**。可使用原生 ANSI 转义代码：`\x1b[1m` 粗体、`\x1b[2m` 暗色、`\x1b[0m` 重置。除非项目已有，否则无需引入样式库。
2. **键盘快捷键**，列于底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。可将按键加粗、描述暗化，或反过来，以清晰为准。

行为：

1. **初始化状态**：一个单一内存对象或结构。启动时渲染第一帧。
2. 每次**读取一个按键（或一行）**，分派给修改状态的处理器。
3. 每次动作后**重新渲染**完整帧，不追加，而是替换。
4. **循环直到退出。**

整个帧应适合一个屏幕。

### 5. 使其可用一个命令运行

向项目现有任务运行器添加脚本（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应运行 `pnpm run <prototype-name>` 或等效命令，绝不需要记住路径。

若宿主项目没有任务运行器，只需将命令放在原型 README 顶部。

### 6. 交给用户

将运行命令交给用户。他们会自行驱动；有趣的时刻是他们说“等等，这不应该可能”或“嗯，我以为 X 会不同”，这些是*想法*中的缺陷，这正是全部目的。若他们希望添加新动作，就添加。原型会演变。

### 7. 捕获答案和原型

原型回答问题后，先捕获答案，再按 [SKILL](SKILL.md) 所述捕获原型。逻辑专属映射：已验证的 reducer、状态机或函数集提取到真实模块中（决策被吸收）；TUI 外壳随一次性分支保留，使原型成为一手来源。

## 反模式

- **不要添加测试。** 需要测试的原型已不再是原型。
- **不要接入真实数据库。** 除非问题专门关于持久化，否则使用内存存储。
- **不要泛化。** 不要问“若之后想支持 X 会怎样”。原型回答一个问题。
- **不要模糊逻辑与 TUI 的界限。** 若 reducer 或状态机引用 `console.log`、提示词或终端转义代码，它就不再可移植。将 TUI 保持为纯模块上的薄外壳。
- **不要将 TUI 外壳交付到生产。** 外壳针对从终端手动驱动而优化，背后的逻辑模块才值得保留。
