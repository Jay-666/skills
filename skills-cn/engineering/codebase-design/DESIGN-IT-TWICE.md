# 设计两次

当用户希望为选定的加深候选项探索替代接口时，使用此并行子 agent 模式。它基于 Ousterhout 的“设计两次”，第一个想法不太可能是最佳方案。

使用 [SKILL.md](SKILL.md) 中的词汇：**模块**、**接口**、**边界点**、**适配器**、**杠杆**。

## 流程

### 1. 确定问题空间

启动子 agent 前，为选定候选项编写面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖的依赖项及其所属类别（参见 [DEEPENING.md](DEEPENING.md)）
- 用于具体化约束的粗略示意代码草图，不是提案，只是使约束具体的方法

向用户展示此内容，然后立即继续第 2 步。子 agent 并行工作时，用户阅读和思考。

### 2. 启动子 agent

使用 Agent 工具并行启动 3 个以上子 agent。每个必须为加深模块产出**截然不同**的接口。

用独立的技术简报提示每个子 agent（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、边界点后有什么）。该简报独立于第 1 步面向用户的问题空间说明。为每个 agent 给出不同设计约束：

- Agent 1：“最小化接口，最多以 1 至 3 个入口点为目标。最大化每个入口点的杠杆。”
- Agent 2：“最大化灵活性，支持多种使用场景和扩展。”
- Agent 3：“为最常见调用方优化，使默认情况变得简单。”
- Agent 4（若适用）：“围绕跨边界点依赖的 ports 和适配器进行设计。”

在简报中同时纳入 [SKILL.md](SKILL.md) 词汇和 CONTEXT.md 词汇，使每个子 agent 的命名符合架构语言和项目领域语言。

每个子 agent 输出：

1. 接口（类型、方法、参数，以及不变量、顺序、错误模式）
2. 展示调用方如何使用的使用示例
3. 实现在边界点后隐藏什么
4. 依赖策略和适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 权衡：哪里杠杆高，哪里薄弱

### 3. 呈现和比较

按顺序呈现设计，让用户吸收每一个，然后用文字比较。按**深度**（接口处的杠杆）、**局部性**（变更集中在哪里）和**边界点位置**进行对比。

比较后，给出自己的建议：你认为哪个设计最强以及原因。若不同设计的元素可良好组合，提出混合方案。应作出明确取舍，用户需要强有力的判断，而非菜单。
