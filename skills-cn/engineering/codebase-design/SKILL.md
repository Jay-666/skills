# 代码库设计

设计**深模块**：在小接口后隐藏大量行为，置于干净的边界处，并可通过该接口测试。在设计或重组代码的任何地方使用这些语言和原则。目标是为调用方提供杠杆，为维护者提供局部性，并让所有人都可测试。

## 术语表

准确使用这些术语，不要以“组件”“服务”“API”或“边界”替代。一致的语言正是目的所在。

**模块**：具有接口和实现的任何事物。有意不限定规模：函数、类、包或跨层切片。_避免使用_：单元、组件、服务。

**接口**：调用方正确使用模块必须知道的一切，包括类型签名，也包括不变量、顺序约束、错误模式、必需配置和性能特征。_避免使用_：API、签名（范围太窄，只指类型级表面）。

**实现**：模块内部的内容，即其代码主体。它不同于**适配器**：一个事物可以是拥有大型实现的小适配器（Postgres 仓储），也可以是拥有小实现的大适配器（内存 fake）。讨论边界时使用“适配器”，否则使用“实现”。

**深度**：接口层面的杠杆，即调用方（或测试）每学习一个单位的接口可运行的行为量。大量行为位于小接口后时模块是**深**的；接口几乎与实现一样复杂时模块是**浅**的。

**边界点** _(Michael Feathers)_：无需在该处编辑即可改变行为的位置，模块接口所处的*位置*。将边界点放在哪里是独立的设计决策，不同于其后隐藏什么。_避免使用_：boundary（与 DDD 的限界上下文多义）。

**适配器**：在边界点处满足接口的具体事物。描述的是*角色*（填补哪个槽位），而非实体（内部有什么）。

**杠杆**：调用方从深度获得的收益，每学习一个单位接口获得更多能力。一个实现可在 N 个调用点和 M 个测试中回报。

**局部性**：维护者从深度获得的收益，变更、缺陷、知识和验证集中在一个位置，而非分散到调用方。修复一次，处处修复。

## 深与浅

**Deep module** = small interface + lots of implementation:

```
┌─────────────────────┐
│   小接口   │  ← 少量方法，简单参数
├─────────────────────┤
│                     │
│  深实现│  ← 隐藏复杂逻辑
│                     │
└─────────────────────┘
```

**浅模块** = 大接口 + 少量实现（应避免）：

```
┌─────────────────────────────────┐
│       大接口           │  ← 多个方法，复杂参数
├─────────────────────────────────┤
│  薄实现            │  ← 仅作转发
└─────────────────────────────────┘
```

设计接口时，询问：

- 能否减少方法数量？
- 能否简化参数？
- 能否在内部隐藏更多复杂性？

## 原则

- **深度是接口属性，而非实现属性。** 深模块的内部可由小型、可 mock、可交换的部件组成，它们只是不属于接口。模块既可在接口处拥有**外部边界点**，也可拥有**内部边界点**（私有于实现，供自身测试使用）。
- **删除测试。** 想象删除模块。若复杂性消失，它只是转发；若复杂性在 N 个调用方中重新出现，它一直在发挥价值。
- **接口是测试表面。** 调用方与测试跨越同一个边界点。若想测试*越过*接口，模块的形态可能有问题。
- **一个适配器意味着假想边界点，两个适配器意味着真实边界点。** 除非确实有跨越它的变化，否则不要引入边界点。

## 为可测试性而设计

好接口使测试自然发生：

1. **接受依赖，不要创建依赖。**

   ```typescript
// 可测试
   function processOrder(order, paymentGateway) {}

// 难以测试
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **返回结果，不要产生副作用。**

   ```typescript
// 可测试
   function calculateDiscount(cart): Discount {}

// 难以测试
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **小表面面积。** 方法越少，所需测试越少；参数越少，测试设置越简单。

## 关系

- 一个**模块**恰好有一个**接口**（呈现给调用方和测试的表面）。
- **深度**是一个**模块**的属性，相对于其**接口**衡量。
- **边界点**是**模块**的**接口**所处的位置。
- **适配器**位于**边界点**并满足**接口**。
- **深度**为调用方产生**杠杆**，为维护者产生**局部性**。

## 已拒绝的表述

- **将深度视为实现行数与接口行数的比率**（Ousterhout）：这会奖励填充实现。我们改用深度即杠杆。
- **将“接口”视为 TypeScript `interface` 关键字或类的公共方法**：范围太窄，此处接口包括调用方必须知道的所有事实。
- **“边界”**：与 DDD 的限界上下文多义。应说**边界点**或**接口**。
