# 加深模块

在已知依赖的情况下，如何安全地加深一组浅模块。假定使用 [SKILL.md](SKILL.md) 中的词汇：**模块**、**接口**、**边界点**、**适配器**。

## 依赖类别

评估加深候选项时，对其依赖进行分类。类别决定如何跨越其边界点测试加深后的模块。

### 1. 进程内

纯计算、内存状态、无 I/O。始终可加深，合并模块并直接通过新接口测试。无需适配器。

### 2. 本地可替代

具有本地测试替身的依赖（Postgres 的 PGLite、内存文件系统）。若替身存在则可加深。测试套件中让替身运行来测试加深后的模块。边界点位于内部，无需在模块外部接口设置 port。

### 3. 远程但自有（Ports 与适配器）

跨网络边界的自有服务（微服务、内部 API）。在边界点定义一个 **port**（接口）。深模块拥有逻辑，传输层作为**适配器**注入。测试使用内存适配器，生产使用 HTTP、gRPC 或队列适配器。

建议形态：*“在边界点定义 port，实现用于生产的 HTTP 适配器和用于测试的内存适配器，使逻辑位于一个深模块中，即使它部署在网络另一端。”*

### 4. 真正外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。加深后的模块将外部依赖作为注入 port 接受，测试提供 mock 适配器。

## 边界点纪律

- **一个适配器意味着假想边界点，两个适配器意味着真实边界点。** 除非至少两个适配器有合理依据（通常是生产和测试），否则不要引入 port。单适配器边界点只是间接层。
- **内部边界点与外部边界点。** 深模块既可在接口处拥有外部边界点，也可拥有内部边界点（私有于实现，供自身测试使用）。不要仅因测试使用内部边界点就通过接口暴露它们。

## 测试策略：替换，而非分层

- 加深模块接口处的测试存在后，浅模块上的旧单元测试就成了冗余，应删除。
- 在加深模块的接口处编写新测试。**接口就是测试表面**。
- 测试应通过接口断言可观察结果，而非内部状态。
- 测试应经受内部重构，它们描述的是行为而非实现。若实现变化时测试必须改变，说明它在测试越过接口的内容。
