# CONTEXT.md 格式

## 结构

```md
# {上下文名称}

{用一两句话说明此上下文是什么，以及为什么存在。}

## 语言

**Order**:
{用一两句话描述该术语}
_避免使用_：采购、交易

**Invoice**:
交付后发送给客户的付款请求。
_避免使用_：账单、付款请求

**Customer**:
下订单的个人或组织。
_避免使用_：客户端、买方、账户
```

## 规则

- **作出明确取舍。** 当同一概念存在多个词时，选择最佳术语，并将其他词列于 `_避免使用_` 下。
- **保持定义简洁。** 最多一两句话。定义它“是什么”，而非“做什么”。
- **仅纳入此项目上下文特有的术语。** 通用编程概念（超时、错误类型、工具模式）即使被项目大量使用也不属于此处。添加术语前问自己：这是此上下文独有的概念，还是通用编程概念？仅前者应被纳入。
- 当自然出现聚类时，**在子标题下分组术语**。若所有术语属于一个内聚区域，使用平铺列表即可。

## 单上下文与多上下文仓库

**单上下文（大多数仓库）：** 在仓库根目录放置一个 `CONTEXT.md`。

**多上下文：** 仓库根目录的 `CONTEXT-MAP.md` 列出各上下文、它们的位置及彼此关系：

```md
# 上下文映射

## 上下文

- [订购](./src/ordering/CONTEXT.md)：接收并跟踪客户订单
- [计费](./src/billing/CONTEXT.md)：生成发票并处理付款
- [履约](./src/fulfillment/CONTEXT.md)：管理仓库拣货和发货

## 关系

- **订购 → 履约**：订购发出 `OrderPlaced` 事件；履约消费它以开始拣货
- **履约 → 计费**：履约发出 `ShipmentDispatched` 事件；计费消费它以生成发票
- **订购 ↔ 计费**：共享 `CustomerId` 和 `Money` 类型
```

此 skill 推断应使用的结构：

- 若存在 `CONTEXT-MAP.md`，读取它以找到各上下文
- 若仅存在根目录 `CONTEXT.md`，则为单上下文
- 若两者均不存在，在第一个术语确定时按需创建根目录 `CONTEXT.md`

存在多个上下文时，推断当前主题与哪个上下文相关。若不清楚则询问。
