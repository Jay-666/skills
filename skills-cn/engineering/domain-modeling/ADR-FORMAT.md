# ADR 格式

ADR 存放于 `docs/adr/`，并使用连续编号：`0001-slug.md`、`0002-slug.md` 等。

按需创建 `docs/adr/` 目录，仅在需要第一个 ADR 时创建。

## 模板

```md
# {决策的简短标题}

{用 1 至 3 句话说明：背景是什么、作出了什么决策，以及原因。}
```

仅此而已。一份 ADR 可以只有一个段落。其价值在于记录*作出了什么决策*和*原因*，而非填满各个章节。

## 可选章节

仅在这些章节确有价值时纳入。大多数 ADR 不需要它们。

- **状态** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`），在重新审视决策时有用
- **已考虑选项**，仅当被拒绝的替代方案值得记住时使用
- **后果**，仅当需要指出不明显的下游影响时使用

## 编号

扫描 `docs/adr/`，找到现有最大编号并加一。

## 何时提出 ADR

以下三项都必须成立：

1. **难以逆转**：未来改变主意的成本很高
2. **脱离上下文会令人困惑**：未来读者看到代码会问“他们为什么要这样做？”
3. **真实权衡的结果**：确实存在替代方案，且你出于特定理由选择其中一个

若一项决策易于逆转，则跳过它，因为你只会直接逆转。若不令人意外，就没人会追问原因。若没有真实替代方案，除了“我们做了显而易见的事”之外，没有内容值得记录。

### 符合条件的内容

- **架构形态。** “我们使用单体仓库。”“写模型采用事件溯源，读模型投影到 Postgres。”
- **上下文之间的集成模式。** “订购和计费通过领域事件通信，而非同步 HTTP。”
- **具有锁定效应的技术选择。** 数据库、消息总线、认证提供方、部署目标。不是每个库，只记录更换需要一个季度的选择。
- **边界和范围决策。** “客户数据由客户上下文拥有，其他上下文仅通过 ID 引用。”明确的“不做什么”与“做什么”同样有价值。
- **有意偏离显而易见的路径。** “由于 X，我们使用手写 SQL 而不是 ORM。”任何合理读者会假定相反做法的事项。它们能阻止下一位工程师“修复”一个有意为之的选择。
- **代码中不可见的约束。** “由于合规要求，不能使用 AWS。”“由于合作方 API 契约，响应时间必须低于 200ms。”
- **拒绝理由不明显的替代方案。** 若曾考虑 GraphQL 但因细微理由选择 REST，记录它，否则六个月后又会有人提出 GraphQL。
