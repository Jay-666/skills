# 领域建模

在设计过程中主动构建并打磨项目的领域模型。这是一项*主动*实践：质疑术语、设计边缘场景，并在术语表和决策明确的当下立即记录。（仅为获得词汇而*阅读* `CONTEXT.md` 并不是此 skill，它只是任何 skill 都可养成的一行习惯。此 skill 用于改变模型，而非仅仅消费模型。）

## 文件结构

大多数仓库只有一个上下文：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

若根目录存在 `CONTEXT-MAP.md`，仓库则有多个上下文。该映射文件指出每个上下文的位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 上下文专属决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

按需创建文件，仅在有内容可写时创建。若不存在 `CONTEXT.md`，在第一个术语被确定时创建它。若不存在 `docs/adr/`，在需要第一个 ADR 时创建它。

## 会话过程中

### 根据术语表提出质疑

当用户使用的术语与 `CONTEXT.md` 中现有语言冲突时，立即指出。“你的术语表将‘取消’定义为 X，但你似乎指的是 Y，究竟是哪一个？”

### 明确模糊语言

当用户使用含糊或多义术语时，提出准确的规范术语。“你说的是‘账户’，是指客户还是用户？它们是不同事物。”

### 讨论具体场景

讨论领域关系时，用具体场景压力测试它们。设计探查边缘情况的场景，促使用户精确界定概念之间的边界。

### 与代码交叉验证

当用户说明某项事物如何工作时，检查代码是否一致。若发现矛盾，提出它：“你的代码会取消整张订单，但你刚才说可以部分取消，哪一种才是正确的？”

### 内联更新 CONTEXT.md

术语确定后，立即更新 `CONTEXT.md`。不要批量处理，应在出现时捕获。使用 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) 中的格式。

`CONTEXT.md` 应完全不包含实现细节。不要将 `CONTEXT.md` 当作规格说明、草稿本或实现决策仓库。它只是术语表，别无其他。

### 谨慎提出 ADR

仅当以下三项都成立时，才提出创建 ADR：

1. **难以逆转**：未来改变主意的成本较高
2. **脱离上下文会令人困惑**：未来读者会问“为什么要这样做？”
3. **真实权衡的结果**：确实存在替代方案，且你出于特定理由选择其中一个

若三者任一缺失，则跳过 ADR。使用 [ADR-FORMAT.md](./ADR-FORMAT.md) 中的格式。
