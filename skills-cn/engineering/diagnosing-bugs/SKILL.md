# 诊断缺陷

一套用于棘手缺陷的实践。仅在有明确理由时跳过阶段。

探索代码库时，阅读 `CONTEXT.md`（若存在）以建立相关模块的清晰心智模型，并检查涉及区域的 ADR。

## 阶段 1：建立反馈循环

**这就是此 skill 的核心。** 其他一切都是机械操作。若你有一个针对该缺陷的**紧凑**通过或失败信号，即能在*这个*缺陷上变红的信号，就能找到原因；二分、假设测试和插桩都只是在消费它。若没有它，盯着代码看再久也无济于事。

在此投入不成比例的精力。**积极、富有创造力，拒绝放弃。**

### 构建方法：大致按此顺序尝试

1. 在能到达缺陷的任意边界处编写**失败测试**：单元、集成或 e2e。
2. 针对运行中开发服务器的 **curl 或 HTTP 脚本**。
3. 使用 fixture 输入的 **CLI 调用**，将 stdout 与已知正确的快照比较。
4. **无头浏览器脚本**（Playwright 或 Puppeteer）：驱动 UI，并对 DOM、控制台或网络进行断言。
5. **重放已捕获的追踪记录。** 将真实网络请求、载荷或事件日志保存到磁盘；在隔离环境中通过代码路径重放。
6. **一次性测试 harness。** 启动系统的最小子集（一个服务和 mock 依赖），通过一次函数调用运行缺陷代码路径。
7. **属性或模糊测试循环。** 若缺陷是“有时输出错误”，运行 1000 个随机输入并查找失效模式。
8. **二分 harness。** 若缺陷出现在两个已知状态之间（提交、数据集、版本），自动化“在状态 X 启动、检查、重复”，从而能执行 `git bisect run`。
9. **差异循环。** 让相同输入通过旧版本和新版本（或两种配置），并比较输出。
10. **HITL bash 脚本。** 最后的手段。若必须由人点击，使用 `scripts/hitl-loop.template.sh` 驱动*人*，使循环仍有结构。捕获输出再反馈给你。

建立正确的反馈循环，缺陷就已经解决了 90%。

### 收紧循环

将循环视为产品。拥有*一个*循环后，进一步**收紧**它：

- 能否让它更快？（缓存设置、跳过无关初始化、缩小测试范围。）
- 能否让信号更清晰？（断言具体症状，而非“未崩溃”。）
- 能否让它更具确定性？（固定时间、设置 RNG 种子、隔离文件系统、冻结网络。）

30 秒且不稳定的循环只比没有循环好一点；2 秒的确定性循环才算紧凑，是调试超能力。

### 非确定性缺陷

目标不是完美复现，而是**提高复现率**。循环触发 100 次、并行化、施压、缩窄时序窗口、注入等待。50% 的偶现缺陷可以调试，1% 的不行，应持续提高复现率直到可调试。

### 确实无法建立循环时

停止并明确说明。列出已经尝试的内容。向用户索取：(a) 可复现问题的环境访问权限，(b) 捕获的产物（HAR 文件、日志转储、核心转储、带时间戳的屏幕录像），或 (c) 添加临时生产插桩的许可。没有循环时**不要**继续提出假设。

### 完成标准：能变红的紧凑循环

当循环既**紧凑**又**具备变红能力**时，阶段 1 才完成：你可以指出**一条已至少运行过一次的命令**，例如脚本路径、测试调用或 curl，并粘贴调用和输出；且该命令：

- [ ] **具备变红能力**：运行实际缺陷代码路径，并断言**用户的确切症状**，因此能在此缺陷上变红、修复后变绿。不是“运行没有报错”，它必须能*捕获这个具体缺陷*。
- [ ] **确定性**：每次运行结论相同（偶现缺陷按上述要求拥有固定且较高的复现率）。
- [ ] **快速**：单位是秒，而非分钟。
- [ ] **可由 agent 运行**：你能无人值守地运行；人仅可通过 `scripts/hitl-loop.template.sh` 参与循环。

若发现自己在这条命令存在前就开始读代码构建理论，**停止：直接跳到假设正是此 skill 要防止的失败。** 没有具备变红能力的命令，就不能进入阶段 2。

## 阶段 2：复现并最小化

运行循环。观察它变红，缺陷出现。

确认：

- [ ] 循环产生的是**用户**描述的失效模式，而不是恰好相邻的其他失败。错误缺陷等于错误修复。
- [ ] 该失败可在多次运行中复现（或对非确定性缺陷，以足够高的频率复现以支持调试）。
- [ ] 已捕获确切症状（错误消息、错误输出、缓慢耗时），以便后续阶段能验证修复确实解决了它。

### 最小化

一旦变红，将复现缩小到**仍会变红的最小场景**。逐次移除输入、调用方、配置、数据和步骤，每次移除后重新运行循环，只保留造成失败所必需的内容。

原因是：最小复现会缩小阶段 3 的假设空间（可疑的活动部件更少），并成为阶段 5 中干净的回归测试。

当**所有剩余元素都必不可少**时完成：移除任意一个都会让循环变绿。

在完成复现**且**最小化前不要继续。

## 阶段 3：提出假设

在测试任何假设前，生成**按优先级排列的 3 至 5 个假设**。仅生成一个假设会锚定在第一个看似合理的想法上。

每个假设都必须**可证伪**：说明它所作出的预测。

> 格式：“若 <X> 是原因，则 <改变 Y> 会使缺陷消失，或 <改变 Z> 会使缺陷变得更糟。”

若无法说明预测，该假设只是感觉，应丢弃或进一步明确。

**在测试前向用户展示优先级列表。** 他们通常拥有能立即重新排序的领域知识（“我们刚部署了与第 3 项有关的变更”），或了解已经排除的假设。这是廉价的检查点，却能节省大量时间。不要因此阻塞，若用户不在则按你的排序继续。

## 阶段 4：插桩

每个探针都必须对应阶段 3 的具体预测。**一次只改变一个变量。**

工具优先级：

1. 环境支持时，使用**调试器或 REPL 检查**。一个断点胜过十条日志。
2. 在能区分假设的边界处使用**目标日志**。
3. 绝不“记录所有内容再 grep”。

为**每条调试日志加上**唯一前缀，例如 `[DEBUG-a4f2]`。最后清理变成一次 grep。未标记的日志会残留，标记日志会被清除。

**性能分支。** 对性能回归，日志通常不合适。应先建立基线测量（计时 harness、`performance.now()`、性能分析器、查询计划），再二分。先测量，再修复。

## 阶段 5：修复并添加回归测试

在修复**之前**编写回归测试，但前提是存在**正确边界**。

正确边界是指测试能按调用点中的实际方式运行**真实缺陷模式**的边界。若唯一可用边界过浅（例如缺陷需要多个调用方却只有单调用方测试，或单元测试无法复现触发缺陷的链路），在此处添加回归测试会带来虚假信心。

**若不存在正确边界，这本身就是发现。** 记录它。代码库架构阻碍了对缺陷的锁定。为下一阶段标记该问题。

若存在正确边界：

1. 将最小复现转化为该边界处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化）场景重新运行阶段 1 的反馈循环。

## 阶段 6：清理与复盘

宣告完成前必须做到：

- [ ] 原始复现不再复现（重新运行阶段 1 循环）
- [ ] 回归测试通过（或已记录不存在边界）
- [ ] 移除所有 `[DEBUG-...]` 插桩（`grep` 该前缀）
- [ ] 删除一次性原型（或移至明确标记的调试位置）
- [ ] 在提交或 PR 消息中写明最终正确的假设，让下一位调试者能够学习

**然后追问：什么能预防此缺陷？** 若答案涉及架构变更（没有好的测试边界、调用方纠缠、隐藏耦合），将具体情况交接给 `/improve-codebase-architecture` skill。在修复完成**之后**而非之前提出建议，因为此时掌握的信息比开始时更多。
