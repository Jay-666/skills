# 询问 Matt

你不必记住每个 skill，因此可以询问。

**流程**是穿过 skills 的路径。大多数路径沿一条**主流程**前进，两个**入口匝道**汇入其中。其余内容则独立存在，或作为底层运行的词汇层。

## 主流程：想法 → 交付

大多数工作经过的路线。你有一个想法，并希望将其构建出来。

1. **`/grill-with-docs`**：通过访谈打磨想法。拥有**代码库**时从这里开始，它是有状态的，会将学到的内容保存在 `CONTEXT.md` 和 ADR 中。（没有代码库？使用 `/grill-me`，见“独立”。两者运行相同的 `/grilling` 原语；`grill-with-docs` 会留下书面记录。）
2. **分支：能在对话中解决每个问题吗？** 若问题需要可运行的答案（状态、业务逻辑、必须看到的 UI），通过原型绕行，并在两个方向都使用 **`/handoff`** 作为桥梁（见“跨会话”）：
   - 使用 **`/handoff`** 输出，再针对该文件打开一个新会话；
   - 使用 **`/prototype`** 以一次性代码回答问题；
   - 使用 **`/handoff`** 将学到的内容交接回去，并从原始想法线程引用它。
3. **分支：这是多会话构建吗？**
   - **是** → **`/to-spec`**（将线程转化为规格说明），再使用 **`/to-tickets`** 拆分为垂直切片任务单，每张任务单都声明其**阻塞依赖**。在本地跟踪器上，`.scratch/<feature>/issues/` 下每张任务单一个文件，人工先处理阻塞项；在真实跟踪器上，依赖边成为原生阻塞链接，任何阻塞项均完成的任务单都可获取。对每张任务单启动 **`/implement`**，并在任务单之间**清空上下文**。
   - **否** → 在此处的同一上下文窗口中直接使用 **`/implement`**。

   无论哪种方式，**`/implement`** 都通过内部驱动 **`/tdd`** 构建每个 issue，一次一个红绿切片；在提交前运行 **`/code-review`**，对差异进行双维度审查（标准 + 规格）。当只希望在没有完整规格的情况下以测试优先构建一个具体行为时，单独使用 **`/tdd`**；当希望相对于固定点审查分支或 PR 时，单独使用 **`/code-review`**。

### 上下文卫生

将第 1 至 3 步保留在**一个不断开的上下文窗口**中，在 `/to-tickets` 之后才压缩或清空，使深入追问、规格说明和任务单都建立在相同思考之上。每个 `/implement` 随后从任务单出发，全新开始。

这一做法的限制是**[智能区](https://www.aihero.dev/ai-coding-dictionary/smart-zone)**，即模型仍能清晰推理的窗口（先进模型约 120k token）。若会话在 `/to-tickets` 前接近此限制，不要在能力退化时硬推，应使用 `/handoff` 并在新线程继续。

## 入口匝道

会生成工作的起始情形，然后汇入主流程。

- **缺陷和请求不断积压** → **`/triage`**。它让 issues 经历分诊角色，产出可供 agent 处理的 issues，之后由 **`/implement`** 获取。

  分诊仅适用于**你没有创建**的 issues：缺陷报告、进入的功能请求及所有原始到达的内容。`/to-tickets` 产出的任务单已经可供 agent 处理，因此**不要对它们分诊**。

- **某些内容坏了** → **`/diagnosing-bugs`**。用于棘手问题：第一眼无法解决的缺陷、间歇性偶现问题、两个已知良好状态之间悄然进入的回归。它在拥有**紧凑反馈循环**前拒绝构建理论，即一条已经能在*这个*缺陷上变红的命令，随后通过回归测试修复。当真正发现是没有好的边界点能锁定缺陷时，它的复盘会交接给 **`/improve-codebase-architecture`**。

- **一个巨大而模糊的工作**，例如绿地项目或超出一个会话容量的大型功能构建 → **`/wayfinder`**，这里认知要求最高的流程。当此处到目标的路径尚不可见时，它在 issue 跟踪器上绘制一张包含**决策任务单**的**共享地图**，逐项解决，产出的是**决策而非交付物**，直至迷雾退去、路径清晰。**`/grill-with-docs`** 打磨可在一个会话中容纳的想法，而路径规划用于无法容纳的想法，速度更慢、密度更高，因此仅用于这种情形，绝不用于范围明确的功能。

  地图清晰后，**它交接，而不构建**：在 **`/to-spec`** 处汇入主流程，将地图中链接的决策收拢为可构建计划，然后照常使用 `/to-tickets` 和 `/implement`。将地图直接接到 `/implement` 会跳过该收拢并丢弃关联细节。仅当工作结果确实很小时才直接进入 `/implement`。

## 代码库健康

不是功能工作，而是维护。

- **`/improve-codebase-architecture`**：只要有空闲时间就运行，使代码库适合 agents 操作。它发现**加深机会**，选择一个会*生成一个想法*，可在 `/grill-with-docs` 处带入主流程。它是寻找候选项的勘察，下面的 **`/codebase-design`** 是设计选中项的工作台。

## 底层词汇

两个由模型调用、在其他 skills *之下*运行的参考，它们各自是词汇的唯一事实来源。当问题在于**词语**而非流程时直接使用它们，或让上方 skills 拉入它们。

- **`/domain-modeling`**：打磨项目的*领域*语言，质疑模糊术语、解决多义词（“账户”承担三个角色），将难以逆转的决策记录为 ADR。这是 `/grill-with-docs` 驱动的主动实践，用于保持 `CONTEXT.md` 是干净的术语表。
- **`/codebase-design`**：用于设计模块*形态*的深模块词汇（模块、接口、深度、边界点、适配器、杠杆、局部性），即在干净边界点的小接口后隐藏大量行为。`/tdd` 和 `/improve-codebase-architecture` 都使用它。

## 跨会话

- **`/handoff`**：当线程已满或需要分叉（例如进入 `/prototype` 会话）时，它将对话压缩为 Markdown 文件。不要原地继续，应**打开一个新会话并引用该文件**以跨越上下文。它是上下文窗口之间双向的桥梁。当你希望拥有**新会话**但需要**保留当前对话**时使用。
- **`/compact`**（内置）：停留在**同一对话**中，让先前轮次被总结。当处于**阶段之间的有意断点**且不介意失去逐字历史时使用。不要在阶段中间压缩，agent 可能迷失方向。`/handoff` 分叉，`/compact` 延续。

## 独立流程

完全脱离主流程。

- **`/grill-me`**：与 `/grill-with-docs` 相同的持续访谈，但适用于**没有代码库**时。它无状态，不在本地保存任何内容，不构建 `CONTEXT.md`。用于打磨不在仓库中的任意计划或设计。
- **`/prototype`**：回答一个设计问题的小型一次性程序，例如此状态模型是否合理，或 UI 应该是什么样子。从第一天起一次性使用，保留答案，删除代码。它是主流程第 2 步的绕行，但任何设计问题难以在纸面上解决时都可使用。
- **`/research`**：将阅读工作委托给**后台 agent**，它根据**一手来源**调查问题，然后在仓库中留下带引用的 Markdown 文件。它阅读时你可以继续工作。产出的文件可以在 `/grill-with-docs` 处带*入*主流程，研究支持思考，而不替代思考。
- **`/teach`**：在多个会话中学习概念，以当前目录作为有状态工作区。
- **`/writing-great-skills`**：用于良好编写和编辑 skills 的参考。

## 前置条件

**`/setup-matt-pocock-skills`**：在第一次工程流程前运行，以配置其他 skills 假定存在的 issue 跟踪器、分诊标签和文档布局。自定义 issue 跟踪器也可使用。
