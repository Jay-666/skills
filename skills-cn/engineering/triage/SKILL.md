# 问题分诊

让项目 issue 跟踪器中的 issues 经过一个小型分诊角色状态机。

若此仓库将外部 pull request 视为请求入口（参见 issue-tracker 配置），分诊也覆盖它们：**PR 是附带代码的 issue**，使用相同角色、状态和状态机，只有少量差异会在下方标为“对于 PR”。根据跟踪器配置将不带前缀的 `#42` 解析为 issue 或 PR。

分诊期间发布到 issue 跟踪器的每条评论或 issue 都**必须**以此免责声明开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md)：如何编写持久的 agent 简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)：`.out-of-scope/` 知识库如何工作

## 角色

两个**类别**角色：

- `bug`：某项内容已损坏
- `enhancement`：新功能或改进

五个**状态**角色：

- `needs-triage`：维护者需要评估
- `needs-info`：等待报告者提供更多信息
- `ready-for-agent`：已完整定义，可供无人值守 agent 处理
- `ready-for-human`：需要人工实现
- `wontfix`：不会处理

对 PR，这些状态针对所附代码解读：`ready-for-agent` 表示已附简报，agent 应对差异采取下一步；`ready-for-human` 表示可由人合并。

每个已分诊 issue 都应恰好带一个类别角色和一个状态角色。若状态角色冲突，标记它，并在做任何事前询问维护者。

这些是规范角色名称，issue 跟踪器使用的实际标签字符串可能不同。应已提供映射；若未提供，运行 `/setup-matt-pocock-skills`。

状态转换：无标签 issue 通常先进入 `needs-triage`，之后可转为 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。报告者回复后，`needs-info` 返回 `needs-triage`。维护者可随时覆盖，若转换看起来异常则标记并在继续前询问。

## 调用方式

维护者调用 `/triage` 并使用自然语言描述所需内容。理解请求并行动。示例：

- “展示需要我注意的任何内容”
- “看看 #42”（issue 或 PR）
- “将 #42 移到 ready-for-agent”
- “有哪些内容可供 agent 获取处理？”

## 展示需要关注的内容

查询 issue 跟踪器，按最旧优先展示三个桶：

1. **无标签**：从未分诊。
2. **`needs-triage`**：正在评估。
3. 自上次分诊备注以来报告者有活动的 **`needs-info`**：需要重新评估。

当 PR 在范围内时，在这些桶中包含外部 PR，并为每一行标记 `[PR]` 或 `[issue]`。发现仅展示*外部* PR（跟踪器配置定义谁属于外部），协作者进行中的 PR 不属于分诊工作。此过滤仅用于发现，明确指定的 PR 无论作者是谁都始终分诊。

展示每项的数量和一行摘要，让维护者选择。

## 分诊指定 issue 或 PR

1. **收集上下文。** 阅读完整 issue 或 PR（正文、评论、标签、作者、日期；对 PR 还要阅读差异）。解析所有既有分诊备注，避免重新询问已解决的问题。使用项目领域术语表探索代码库，并遵守涉及区域的 ADR。对代码库执行两项检查：(a) **冗余**，按领域概念（而非仅请求措辞）搜索所请求行为的现有实现，并报告查找位置。若找到，它是已实现的 `wontfix`（第 5 步）。(b) **先前拒绝**，读取 `.out-of-scope/*.md` 并指出所有与此请求相似的内容。

2. **建议。** 向维护者说明你的类别和状态建议及理由，并提供与请求相关的简短代码库摘要，包括是否已实现。等待指示。

3. **验证主张。** 在任何深入追问前，检查主张是否成立。对缺陷，按报告者步骤复现。对 PR，确认差异确实完成其声称的内容，检出它并运行相关测试或命令。报告结果：已确认（附代码路径）、失败或细节不足（强烈的 `needs-info` 信号）。经确认的验证可形成更强的 agent 简报。

4. **深入追问（若需要）。** 若请求需要补充细节，同时运行 `/grilling` 和 `/domain-modeling` skills，逐个问题地把它塑造成形，在决策落定时明确领域术语并内联更新 `CONTEXT.md` 或 ADR。

5. **应用结果：**
   - `ready-for-agent`：发布 agent 简报评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human`：与 agent 简报结构相同，但说明为何不可委派（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info`：发布分诊备注（模板见下）。
   - `wontfix`：关闭，评论取决于*原因*：
     - **已实现**：变更已存在于代码库。指出其位置，**不要**写入 `.out-of-scope/`（该知识库用于*被拒绝*请求，而非已构建内容）。
     - **已拒绝（缺陷）**：礼貌说明后关闭。
     - **已拒绝（增强）**：写入 `.out-of-scope/`，从评论链接它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage`：应用该角色。若已有部分进展，可选发布评论。

## 快速状态覆盖

若维护者说“将 #42 移到 ready-for-agent”，相信他们并直接应用该角色。确认即将执行的操作（角色变更、评论、关闭），然后行动。跳过深入追问。若未经过深入追问会话就移至 `ready-for-agent`，询问是否希望编写 agent 简报。

## Needs-info 模板

```markdown
## 分诊备注

**目前已确定的内容：**

- 要点 1
- 要点 2

**我们仍需要你（@reporter）提供的内容：**

- 问题 1
- 问题 2
```

在“目前已确定的内容”下记录深入追问期间解决的一切，避免工作丢失。问题必须具体且可行动，而非“请提供更多信息”。

## 恢复先前会话

若 issue 或 PR 上存在先前分诊备注，阅读它们，检查报告者是否回答了未决问题，并在继续前呈现更新后的图景。不要重新询问已解决的问题。
