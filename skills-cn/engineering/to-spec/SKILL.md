此 skill 根据当前对话上下文和对代码库的理解生成规格说明（也可称为 PRD）。**不要**访谈用户，只需综合已经掌握的信息。

应已提供 issue 跟踪器及分诊标签词汇；若尚未提供，运行 `/setup-matt-pocock-skills`。

## 流程

1. 若尚未完成，探索仓库以了解代码库的当前状态。在整份规格说明中使用项目领域术语表的词汇，并遵守涉及区域的所有 ADR。

2. 勾勒测试该功能时使用的边界。优先采用现有边界，并选择尽可能高层的边界。若需要新边界，在可行的最高层提出它。整个代码库中的边界越少越好，理想数量是一个。

与用户确认这些边界符合其预期。

3. 使用下方模板编写规格说明，然后发布到项目 issue 跟踪器。添加 `ready-for-agent` 分诊标签，无需额外分诊。

<spec-template>

## 问题陈述

从用户视角描述其面临的问题。

## 解决方案

从用户视角描述问题的解决方案。

## 用户故事

一份很长的、按编号排列的用户故事列表。每个用户故事应采用以下格式：

1. 作为<角色>，我希望<功能>，以便<收益>

<user-story-example>
1. As a mobile bank customer, I want to see balance on my accounts, so that I can make better informed decisions about my spending
</user-story-example>

这份用户故事列表应非常全面，覆盖该功能的所有方面。

## 实现决策

列出已作出的实现决策，可包括：

- 将要构建或修改的模块
- 将要修改的模块接口
- 开发者给出的技术澄清
- 架构决策
- Schema 变更
- API 契约
- 具体交互

**不要**包含具体文件路径或代码片段，它们很快可能过时。

例外情况：若原型产出了比文字更精确地表达决策的片段（状态机、reducer、schema、类型形状），可将其内嵌在相关决策中，并简要注明其来自原型。仅保留信息密度高的决策部分，不要放入可运行演示。

## 测试决策

列出已作出的测试决策，包括：

- 好测试的定义（仅测试外部行为，而非实现细节）
- 将要测试的模块
- 测试的既有范例（即代码库中的相似测试）

## 范围外内容

说明不属于本规格说明范围的内容。

## 补充说明

有关该功能的其他说明。

</spec-template>
