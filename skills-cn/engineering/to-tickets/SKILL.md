# 转为任务单

将计划、规格说明或对话拆分为一组**任务单**，即垂直切片；每项任务单都声明**阻塞**它的其他任务单。

应已提供 issue 跟踪器及分诊标签词汇；若尚未提供，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集上下文

基于对话上下文中已有的信息工作。若用户以参数传入引用（规格说明路径、issue 编号或 URL），获取它并阅读完整正文和评论。

### 2. 探索代码库（可选）

若尚未探索代码库，则先了解代码的当前状态。任务单标题和描述应使用项目领域术语表词汇，并遵守涉及区域的 ADR。

寻找通过预重构使实现更容易的机会。“先让变更变得容易，再完成这个容易的变更。”

### 3. 起草垂直切片

将工作拆分为**垂直切片**任务单。

<vertical-slice-rules>

- 每个切片都贯穿各层（schema、API、UI、测试）形成狭窄但**完整**的路径，是垂直切片，而非单层的水平切片
- 完成的切片可单独演示或验证
- 每个切片的规模应能放入单个全新的上下文窗口
- 所有预重构应先完成

</vertical-slice-rules>

为每项任务单给出其**阻塞依赖**，即必须先完成的其他任务单。没有阻塞项的任务单可以立即开始。

**宽范围重构是垂直切片的例外。** **宽范围重构**是单项机械变更，例如重命名列、重新标注共享符号类型，但其**影响范围**扩散至整个代码库，一次编辑会破坏数千个调用点，任何垂直切片都无法独立保持通过。不要强行将它拆成垂直切片；应按**扩展-收缩**排序。先扩展：在旧形式旁添加新形式，确保没有内容损坏。再按影响范围划分批次迁移调用点（按包或目录），每个批次独立成任务单并依赖扩展任务；因为旧形式仍存在，批次之间可保持 CI 通过。最后收缩：在没有调用方后删除旧形式，该任务单依赖全部迁移批次。当各批次也无法单独保持通过时，保留该顺序，但让它们共享一个集成分支，并共同阻塞最终的集成和验证任务单；只承诺在最终任务单中通过。

### 4. 向用户确认

将建议的拆分方案以编号列表呈现。每项任务单展示：

- **标题**：简短的描述性名称
- **受阻于**：必须先完成的其他任务单（如有）
- **交付内容**：此任务单使其正常工作的端到端行为

询问用户：

- 粒度是否合适？（过粗 / 过细）
- 阻塞依赖是否正确？每项任务单是否仅依赖真正制约它的任务单？
- 是否应合并某些任务单或进一步拆分？

迭代直至用户批准拆分方案。

### 5. 将任务单发布到已配置的跟踪器

发布已批准的任务单。具体**方式**取决于 `/setup-matt-pocock-skills` 配置的跟踪器：任务单本身相同，只有阻塞依赖的表现形式不同：

- **本地文件**：在 `.scratch/<feature-slug>/issues/<NN>-<slug>.md` 下为每项任务单写入一个文件，按依赖顺序从 `01` 编号（阻塞项在前）。每个文件的“受阻于”列出其依赖的编号或标题。使用下方逐任务单文件模板，每个文件仅放一项任务单，绝不合并为一个文件。
- **真实 issue 跟踪器（GitHub、Linear 等）**：按依赖顺序（阻塞项在前）为每项任务单发布一个 issue，以便任务单的阻塞依赖可引用真实标识符。平台有原生阻塞或子 issue 关系时使用它；否则将每项任务单的“受阻于”设为阻塞它的 issues。除非另有指示，添加 `ready-for-agent` 分诊标签，因为这些任务单本就可由 agent 获取处理。

处理**前沿任务**：即所有阻塞项均已完成的任务单。对于纯线性链，这意味着从上到下依次处理。

**不要**关闭或修改任何父 issue。

无论使用哪种形式，都避免具体文件路径或代码片段，因为它们很快会过时。例外情况：若原型产出了比文字更精确地表达决策的片段（状态机、reducer、schema、类型形状），可内嵌该片段并简要注明它来自原型。仅保留信息密度高的决策部分，而非可运行的演示。

使用 `/implement` 一次处理一项前沿任务单，并在任务单之间清空上下文。
