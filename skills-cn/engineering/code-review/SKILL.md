对 `HEAD` 与用户提供固定点之间的差异进行双维度审查：

- **标准**：代码是否符合本仓库记录的编码标准？
- **规格**：代码是否忠实实现了原始 issue、PRD 或规格说明？

两个维度均作为**并行子 agent** 运行，避免相互污染上下文，之后由此 skill 汇总其发现。

应已提供 issue 跟踪器；若不存在 `docs/agents/issue-tracker.md`，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 确定固定点

用户所说的内容就是固定点，例如提交 SHA、分支名、标签、`main`、`HEAD~5` 等。若未指定，则询问。

记录一次差异命令：`git diff <fixed-point>...HEAD`（三点形式，因此与合并基点比较）。同时通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

继续前，确认固定点可以解析（`git rev-parse <fixed-point>`）且差异非空。错误引用或空差异应在此失败，而非在两个并行子 agent 内部失败。

### 2. 确定规格来源

按以下顺序查找原始规格说明：

1. 提交信息中的 issue 引用（`#123`、`Closes #45`、GitLab `!67` 等），通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户以参数传入的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD 或规格文件。
4. 若未找到，询问用户规格说明所在位置。若用户表示没有，**规格**子 agent 跳过并报告“无可用规格说明”。

### 3. 确定标准来源

仓库中所有记录代码编写方式的内容，例如 `CODING_STANDARDS.md` 或 `CONTRIBUTING.md`。

除仓库记录的内容外，标准维度始终包含下面的**异味基线**：一组固定的 Fowler 代码异味（《重构》第 3 章），即使仓库没有记录任何标准也适用。它受两条规则约束：

- **仓库优先。** 已记录的仓库标准始终优先；若它认可基线会标记的内容，则抑制该异味。
- **始终属于判断。** 每种异味都是带标签的启发式判断（“可能存在特性依恋”），而非硬性违规；与此处任何标准一样，跳过工具已强制执行的内容。

每种异味按“*是什么* → *如何修复*”描述，将其与差异进行匹配：

- **神秘命名**：函数、变量或类型名称未揭示其用途或所持内容。→ 重命名；若找不到诚实的名称，说明设计含糊。
- **重复代码**：相同逻辑形态出现在变更的多个块或文件中。→ 提取共享形态，并从两处调用。
- **特性依恋**：一个方法访问另一对象的数据多于自身数据。→ 将方法移到其依恋的数据上。
- **数据泥团**：相同的少量字段或参数反复一起传递（一个等待诞生的类型）。→ 将它们打包为一个类型并传递该类型。
- **基本类型偏执**：用基本类型或字符串代替值得拥有独立类型的领域概念。→ 为该概念创建自己的小类型。
- **重复的条件分支**：针对同一类型的相同 `switch` 或 `if` 级联在变更中重复出现。→ 用多态替换，或让两个位置共享同一个映射。
- **散弹式修改**：一项逻辑变更迫使在差异的许多文件中分散编辑。→ 将共同变化的内容聚集到一个模块中。
- **发散式变化**：一个文件或模块因多个无关原因被编辑。→ 拆分，使每个模块仅因一个原因变化。
- **投机性通用**：为规格并无需求而添加的抽象、参数或钩子。→ 删除它，在出现真实需求前内联回去。
- **消息链**：调用方不应依赖的长链式导航 `a.b().c().d()`。→ 在第一个对象上用一个方法隐藏这段遍历。
- **中间人**：一个类或函数主要只是继续委托。→ 删除它，直接调用真正目标。
- **被拒绝的遗赠**：子类或实现者忽略或重写继承内容的大部分。→ 放弃继承，改用组合。

### 4. 并行启动两个子 agent

发送一条包含两个 `Agent` 工具调用的消息。两个调用均使用 `general-purpose` 子 agent。

**标准子 agent 提示词**，应包含：

- 完整差异命令和提交列表。
- 第 3 步找到的标准来源文件列表，**加上第 3 步的异味基线全文**。子 agent 无法以其他方式访问它。
- 简述：“报告与每个相关文件或块对应的：(a) 差异违反已记录标准的所有位置，引用标准（文件和规则）；(b) 发现的任何基线异味，给出名称并引用块。区分硬性违规与判断：已记录标准的违反可以是硬性违规，但基线异味始终是判断，且已记录的仓库标准覆盖基线。跳过工具已强制执行的内容。限 400 词以内。”

**规格子 agent 提示词**，应包含：

- 差异命令和提交列表。
- 规格说明的路径或已获取的内容。
- 简述：“报告：(a) 规格要求但缺失或仅部分完成的需求；(b) 差异中未被要求的行为（范围蔓延）；(c) 看似已实现但实现可能错误的需求。每项发现均引用规格说明行。限 400 词以内。”

若缺少规格说明，跳过规格子 agent，并在最终报告中注明。

### 5. 汇总

在 `## 标准` 和 `## 规格` 标题下逐字呈现或轻度清理两个报告。**不要**合并或重新排序发现，两个维度有意保持分离（见“为何使用两个维度”）。

以一行摘要结尾：每个维度的发现总数及其*维度内*最严重的问题（若有）。不要跨维度选出单一最严重问题，这正是分离要避免的重新排序。

## 为何使用两个维度

一项变更可以通过一个维度而在另一个维度失败：

- 遵循所有标准但实现了错误内容的代码 → **标准通过，规格失败。**
- 精确完成 issue 要求但破坏项目约定的代码 → **规格通过，标准失败。**

分开报告可避免一个维度掩盖另一个维度。
